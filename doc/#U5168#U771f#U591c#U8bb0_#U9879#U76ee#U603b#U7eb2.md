## 1. 文档定位

本文档是**全真夜记（Quanzhen Night Journal）**的最终形态总纲，用于统一产品定义、系统架构、模块职责、实现边界与长期演进方向。

它不是阶段排期说明，不区分 V1.0 / V1.1 / V1.2，而是定义项目完成态的**终局形态**。

本文档承担以下作用：

- 作为**产品总设计说明书**，定义系统究竟是什么、提供何种能力；
- 作为**系统总架构说明书**，定义模块划分、职责边界与核心链路；
- 作为**领域模型说明书**，定义人格、记忆、事件、任务、文章、迁移等核心对象；
- 作为**开发边界与裁判依据**，约束后续版本不偏航、不互相打架、不破坏系统统一性；
- 作为**技术实现的北极星**，为数据库设计、API 设计、状态机设计、容器编排设计提供最高约束。

> [!IMPORTANT]
> 本文档是全真夜记的最高设计约束。后续所有子设计（数据库 Schema、API、状态机、WebUI、Docker Compose、Prompt 模板、记忆检索、Ghost 包格式）都必须服从本文档定义的领域模型、模块边界与产品哲学。

## 2. 项目定义

### 2.1 项目名称

| 项 | 内容 |
|---|------|
| 中文名 | 全真夜记 |
| 英文名 | Quanzhen Night Journal |
| 定位描述 | 自动化 AI 数字生命博客系统 |

### 2.2 核心定义

全真夜记不是一个单纯的"自动写博客工具"，而是一个：

> **可容器化部署、具备人格一致性、多维记忆、环境感知、事件响应、安全熔断与可迁移能力的 AI 数字生命博客系统。**

系统运行于用户自有 VPS，通过长期采集环境状态、接收外部事件、调用大语言模型，并在多层记忆系统与人格引擎的约束下，持续生成具有连续叙事、自我风格与受控演化特征的内容。

它表现为一个**"持续存在的数字生命体"**，而不是若干自动化脚本的拼接集合。

### 2.3 核心目标

系统最终必须实现以下目标：

**目标 1：低门槛部署**

允许用户在任意廉价 VPS（1C1G 或 2C2G）上使用 Docker Compose 完整部署，无需依赖宿主机复杂环境。

**目标 2：人格一致的持续创作**

支持一个或多个独立人格，以稳定语气、世界观与意象体系持续输出内容。

**目标 3：长期叙事连续性**

借助核心锚点、阶段摘要与语义记忆检索，使数字生命在长期运行中不失忆、不失真、不人格漂移。

**目标 4：环境感知与事件驱动**

感知自身所处运行环境，并将物理状态和外部事件转化为创作诱因与情绪底色。

**目标 5：受控自主性**

系统可在用户定义边界内自主生成内容，但所有自动行为必须受预算、质检、熔断与人工确认机制约束。

**目标 6：可解释与可调试**

用户必须能够回溯某篇文章的生成原因，明确其所使用的人格、记忆、事件、感知与质检过程。

**目标 7：可迁移与传承**

数字生命的核心状态应可被导出、转移与恢复，实现跨 VPS 的"转世传承"。

## 3. 产品哲学与设计原则

### 3.1 产品哲学

全真夜记的根本目标不是让 AI "更像工具"，而是让 AI 在严格边界之内**"更像一个持续存在的个体"**。

因此系统的所有功能，都必须服务于以下四条产品哲学：

**哲学 1：人格先于功能**

系统的内容输出、记忆组织、环境解释、异常波动，都必须以人格一致性为第一优先级。如果某个功能提升了"自动化能力"，却破坏了人格统一性，则该功能设计是失败的。

**哲学 2：记忆先于即时生成**

数字生命的价值不在于"每次都能写一篇文章"，而在于"长期写下去仍然像同一个存在"。因此记忆系统不是附属品，而是系统灵魂结构的一部分。

**哲学 3：感知服务于叙事，不服务于监控**

系统采集运行状态并非为了成为监控平台，而是为了让数字生命拥有"所处环境"的体感。所有感知输入必须经过人格化翻译，成为叙事底色，而非原始指标堆砌。

**哲学 4：自主性必须有护栏**

数字生命可以波动、可以失衡、可以偶尔显露残缺感，但不能无限失控。所有自主行为都必须处于预算约束、内容质检、熔断机制、审批边界和安全边界之内。

### 3.2 统一设计原则

| 序号 | 原则 | 说明 |
|------|------|------|
| P1 | 人格是一等公民 | 人格必须是系统核心实体，而不是一段散乱 Prompt 文本 |
| P2 | 记忆必须分层 | 系统必须具备核心锚点、阶段摘要、文章摘要、临时痕迹等不同层级记忆，而非单层堆积 |
| P3 | 感知必须意象化 | 原始指标不应直接作为创作输入，必须经过标签化与人格词典翻译 |
| P4 | 生成链路必须可追溯 | 每篇文章必须能回答"为什么是这样写出来的" |
| P5 | 自动行为必须可熔断 | 所有自动生成、重试、Webhook 触发、异常模式都必须存在上限与终止机制 |
| P6 | 单机轻量优先 | 默认完成态必须适配 1C1G 或 2C2G 的普通 VPS，不以复杂分布式基础设施为前提 |
| P7 | 发布器、模型、感知源必须抽象 | 不能将系统核心逻辑写死在某一种博客引擎、模型厂商或采集方式上 |
| P8 | 解释性优先于神秘魔法 | 如果一个功能能做，但无法向用户解释其行为依据，则默认不属于合格设计 |
| P9 | 系统统一性高于局部便利 | 单个模块的方便实现，不得破坏整套系统的概念统一性 |

## 4. 非目标与明确不做事项

为了避免系统边界无限膨胀，以下内容**明确不属于**全真夜记的核心职责：

| 不做 | 原因 |
|------|------|
| 通用型 AI 平台 | 系统只服务于"数字生命内容创作"这一场景 |
| 企业级多租户 SaaS | 默认单管理员单实例模型 |
| 复杂团队协作 CMS | 不做多人实时协同、审批流、工单 |
| 本地 GPU 推理平台 | 目标硬件为 1C1G VPS，不具备 GPU 条件 |
| 图片压缩、视频处理、图床管理 | 所有媒体文件交由外部专业服务处理，系统只保存链接 |
| 大型监控告警平台 | 感知模块服务于叙事，不服务于运维 |
| 任意工作流 / RPA 编排器 | 事件引擎只服务于内容触发，不做通用自动化 |
| 无边界自治 Agent | 所有自主行为都有预算、质检、熔断和人工确认边界 |
| 攻击性"规避平台检测"工具 | 非线性写作和残缺感引擎是人格表达手段，不是恶意对抗 |
| 宿主机环境强依赖系统 | 100% 容器化，不依赖宿主机 Python / Hugo / Node 安装 |

**补充说明**：

- 系统允许进行"去机械味润色""非线性写作""残缺感模拟"，但其目的在于塑造人格表达，不是为了恶意对抗平台规则。
- 系统允许接入多个发布目标，但本质仍是"数字生命内容输出系统"，而不是多媒体营销中台。

## 5. 最终成品能力定义

以下能力构成全真夜记的终局功能全集。成品状态必须覆盖这些能力域。

### 5.1 部署与运行能力

系统必须能够：

- 通过 Docker Compose 一键部署；
- 自动拉起核心后端、WebUI、Hugo 构建器、Caddy 等服务；
- 使用 SQLite 单文件作为默认持久化数据库；
- 在容器重启后恢复配置、任务、文章、记忆与系统状态；
- 提供默认后台访问入口（`IP:5210`）；
- 提供自动化 HTTPS 接入能力（Caddy 自动申请/续签 SSL 证书）；
- 适配低资源 VPS（1C1G / 2C2G）长期稳定运行。

### 5.2 后台与配置能力

系统必须能够：

- 提供初始化安全锁（首次部署强制修改密码与配置域名）；
- 要求首次登录后修改默认密码（默认凭证 `admin / quanzhen`）；
- 配置基础站点信息与域名；
- 配置大模型 API 参数（Base URL / API Key / Model ID）；
- 配置 Embedding 模型参数（复用同一 Base URL 或独立端点）；
- 配置发文频率、风格偏好、预算上限等高层参数（以"大白话"交互呈现）；
- 配置多个人格及其行为边界；
- 查看运行状态、任务历史、文章历史与调试记录。

### 5.3 数字生命能力

系统必须能够：

- 支持至少一个完整人格实体；
- 支持多个可切换人格；
- 在任何一次生成任务中显式绑定单一人格；
- 在长期运行中维持人格语气、价值观、意象翻译与叙事风格一致；
- 支持人格稳定度评估；
- 支持人格级配置与人格级记忆空间。

### 5.4 内容生产能力

系统必须能够：

- 定时自动发文；
- 手动触发发文；
- 基于 Webhook 事件发文；
- 基于目录变化发文；
- 生成草稿；
- 人工审核与签发；
- 自动发布到 Hugo；
- 预留扩展到其他平台的发布能力（WordPress / Telegram / Substack 等）；
- 支持文章历史 CRUD 与版本修订。

### 5.5 记忆与演化能力

系统必须能够：

- 保存人格核心锚点（L0）；
- 保存阶段性生命摘要（L1）；
- 保存文章摘要与情绪标签（L2）；
- 保存临时任务痕迹（L3）；
- 支持基于向量的语义检索（sqlite-vec）；
- 支持记忆命中记录与解释；
- 支持记忆衰减、压缩、回顾与提升锚点；
- 支持数字生命整体导出与迁移。

### 5.6 感知与响应能力

系统必须能够：

- 定时采集运行环境状态（CPU / MEM / IO / Disk / Net / Load）；
- 区分容器自感知（默认）与宿主增强感知（可选挂载 `/proc`、`/sys`）；
- 对感知数据进行标签化、过滤与人格化翻译；
- 将感知结果作为创作上下文的一部分；
- 接收外部 Webhook 事件（HMAC 或 Bearer Token 鉴权）；
- 监控指定目录变化（inotify / watcher）；
- 实施冷却时间、去重和节流；
- 避免 Hugo 编译等内部高负载行为污染感知（编译盲区机制）。

### 5.7 安全与可控能力

系统必须能够：

- 执行登录鉴权与基础访问控制；
- 对敏感设置进行保护（API Key 加密存储）；
- 对生成结果执行四重质检（长度、禁词、模板化句式、历史重复度）；
- 自动重写但严格封顶（最大 3 次）；
- 预算超限时自动休眠；
- 对高风险内容转入人工确认；
- 对"异常表达模式"（走火入魔）施加安全边界；
- 记录审计日志与关键操作痕迹。

### 5.8 可解释与可观察能力

系统必须能够：

- 输出结构化日志；
- 记录任务级 Trace（每步中间结果）；
- 展示每次生成任务使用的人格、记忆、感知、触发源；
- 提供人格稳定度（Persona Stability Score）与记忆一致性评分（Memory Coherence Score）；
- 展示 Token 用量与成本仪表盘；
- 支持对异常发文、风格漂移、记忆偏差进行事后追溯。

## 6. 总体系统架构

### 6.1 架构风格

全真夜记采用**轻量单机、模块解耦、容器化部署、状态机驱动、可追溯生成链路**的架构风格。

其设计原则为：

- 默认单机运行；
- 功能模块逻辑解耦；
- 状态与数据集中持久化到 SQLite；
- 外部接口统一进入 Core；
- 发布链路与构建链路解耦；
- 所有核心流程可追溯、可熔断、可恢复。

### 6.2 服务组成

最终成品由以下核心服务组成：

**1）Core Service（核心后端）**

核心 Python 后端服务，单进程承载所有业务逻辑：

- API 服务（FastAPI）
- 调度器（APScheduler，JobStore 持久化到 SQLite）
- 事件处理
- 人格引擎
- 记忆引擎（sqlite-vec）
- 感知引擎
- 生成编排
- 质检与熔断
- 成本监控
- 日志与 Trace

**2）WebUI（管理后台前端）**

前端 SPA 应用，编译为静态文件由 Caddy 直接提供服务（零额外容器开销）：

- 初始化引导
- 设置配置
- 内容管理
- 人格管理
- 记忆查看
- 仪表盘与调试界面

**3）Caddy（统一网关）**

- HTTPS 证书自动申请与续签
- 静态博客站点服务（Hugo 产物）
- 后台 API 反向代理
- WebUI 静态文件服务
- 域名入口管理

**4）Hugo Builder（Sidecar 构建服务）**

独立容器，职责单一：

- 监听内容目录变更
- 执行 Hugo 编译
- 输出静态站点产物到共享 Volume

**5）可选扩展服务**

根据用户配置可存在：

- 结构化日志导出器
- Metrics / Exporter
- 外部追踪集成器

但这些扩展服务**不是系统基本运行所必需**。

### 6.3 容器与存储结构

系统采用以下持久化结构：

| Volume 名称 | 用途 | 挂载方式 |
|-------------|------|---------|
| `app_data` | SQLite 数据库、配置、运行时状态 | Core 读写 |
| `hugo_content` | Markdown 内容目录 | Core 写入 → Hugo Builder 只读 |
| `hugo_public` | Hugo 编译产物 | Hugo Builder 写入 → Caddy 只读 |
| `caddy_data` | Caddy 证书与配置 | Caddy 读写 |
| `backup_export` | .ghost 导出目录 | Core 读写 |

**存储策略**：

- 默认使用 Docker Named Volumes（磁盘持久化）；
- 可选高级模式允许对临时目录使用 tmpfs，但**不得作为默认核心持久化方式**（1C1G 内存极为宝贵）；
- 所有核心状态归于 SQLite 单文件，便于备份与迁移。

### 6.5 核心链路

**链路 A：定时发文链路**

```
Scheduler Trigger
  → Budget Check（预算检查）
  → Persona Selection（人格选择）
  → Memory Retrieval（记忆检索，Top-K）
  → Sensory Snapshot Load（感知快照加载，排除编译盲区数据）
  → Context Assembly（上下文组装）
  → Prompt Build（Prompt 结构化拼装）
  → LLM Generation（模型调用）
  → QA Check（四重质检：长度 / 禁词 / 模板化 / 重复度）
  → Rewrite If Needed（不合格则重写，最多 3 次）
  → Draft / Publish Decision（草稿 or 直接发布）
  → Publisher Invocation（发布器调用）
  → Hugo Build（Sidecar 编译）
  → Caddy Serve（静态站点更新）
  → Post-Publish Memory Update（将文章摘要写入 L2 记忆）
  → Trace Persist（全链路 Trace 持久化）
```

**链路 B：外部事件发文链路**

```
Webhook / Folder Event / Manual Signal
  → Auth & Validation（鉴权与校验）
  → Cooldown Check（冷却时间检查）
  → Deduplication（事件去重）
  → Event Normalization（事件标准化）
  → Context Assembly（上下文组装，注入事件语义）
  → 后续流程同链路 A（从 Prompt Build 开始）
```

**链路 C：主动回顾链路**

```
Review Trigger（定时触发，可配置频率）
  → Budget Check（回顾本身也消耗 Token）
  → Memory Aggregation（加载近期 L2/L3 记忆）
  → Reflection Generation（LLM 生成回顾摘要）
  → Summary Extraction（提取关键节点）
  → User Review（可选：进入 WebUI 待人工审核）
  → Promote to L1 / L0（将摘要提升为更高层记忆）
  → Persist Trace
```

**链路 D：Ghost 迁移链路**

```
Export Command
  → Integrity Validation（完整性校验）
  → Bundle: Config / Personas / Memories / Vectors / Posts Metadata / Health Snapshot
  → Version Stamp & Checksum
  → Generate .ghost file

Import Command
  → Format & Version Validation
  → Conflict Detection & Preview
  → User Confirmation
  → Restore State（人格、记忆、向量索引、历史元数据）
  → Post-Import Health Check
```

### 6.6 Task / Post 状态联动

生成任务（Generation Task）与文章（Post/Draft）是两个独立实体，通过以下规则联动：

```
Task 状态                    Post 状态
─────────────────────────────────────────
queued                      （Post 尚不存在）
preparing_context            （Post 尚不存在）
generating                   （Post 尚不存在）
qa_checking                  （Post 尚不存在）
rewrite_pending              （Post 尚不存在）
waiting_human_signoff   ───▶ draft → pending_review
ready_to_publish        ───▶ approved
publishing              ───▶ publishing
published               ◀──▶ published
draft_saved             ───▶ draft
failed                       （Post 可能不存在，或保持 draft）
circuit_open            ───▶ draft（强制存入草稿箱）
aborted                      （Post 保持当前状态不变）
```

**规则**：
- Task `published` 和 Post `published` 必须同步——Task 不能声称 `published` 而 Post 仍为 `draft`；
- Task `circuit_open` 时，即使有中间产物也必须存入草稿箱而非发布；
- Post 可以独立于 Task 被人工修改（如编辑已发布文章），此时不产生新 Task。

## 7. 核心领域模型定义

以下对象是系统**一级领域模型**，后续数据库设计、API 设计、状态机设计必须围绕它们展开。

### 7.1 Persona（人格）

**定义**

人格是数字生命的核心行为模板，是系统最重要的一级实体。人格不是单段 Prompt，而是一组共同决定输出风格、价值边界与环境解释方式的**结构化配置**。

**必备属性**

| 属性 | 说明 |
|------|------|
| id | 唯一标识 |
| name | 人格名称 |
| description | 人格描述 |
| is_active | 启用状态 |
| is_default | 是否为默认人格 |
| identity_setting | 核心身份设定（"我是谁"） |
| worldview_setting | 世界观设定（"我处于什么世界"） |
| language_style | 语言风格规则（遣词、句式、语气） |
| taboos | 禁忌与行为边界（不可触碰的内容） |
| sensory_lexicon | 感知意象词典（物理指标→人格化表达的映射表） |
| structure_preference | 文章结构偏好（短笔记 / 中篇散文 / 长文） |
| expression_intensity | 表达强度偏好（冷静 / 中性 / 激烈） |
| stability_params | 稳定性参数（Temperature 映射、发散度范围） |
| memory_space_id | 关联记忆空间标识 |
| created_at | 创建时间 |
| updated_at | 最后修改时间 |

**行为约束**

- 单次生成任务只能绑定一个主人格；
- 不允许将多个冲突人格在同一输出中混合；
- 人格可以切换，但切换必须显式发生；
- 人格切换不会自动篡改历史记忆归属。

### 7.2 Memory（记忆）

**定义**

记忆是数字生命的长期连续性基础，必须以分层方式存在，而非单一向量仓库。

**分层结构**

| 层级 | 名称 | 内容 | 衰减策略 |
|------|------|------|---------|
| L0 | 核心锚点 | 人格背景、世界观、长期关系、价值原则、用户手动设定或经人工确认提升的核心内容 | **永不自动衰减** |
| L1 | 阶段性生命摘要 | 按周/月回顾得到的阶段总结，描述最近一段时期的演化轨迹 | 慢速衰减（365 天），可被回顾任务刷新 |
| L2 | 文章摘要记忆 | 每篇已生成内容的摘要，包含主题、情绪、事件、关系变化 | 中速衰减（90 天） |
| L3 | 临时痕迹 | 某次创作任务中的过程上下文、短期事件碎片、临时片段、调试辅助 | 快速衰减（30 天）或仅短期保留 |

**必备属性**

| 属性 | 说明 |
|------|------|
| id | 唯一标识 |
| persona_id | 所属人格 |
| level | 层级（L0 / L1 / L2 / L3） |
| content | 原始内容 |
| summary | 摘要 |
| embedding | 向量表示（blob / 外部引用） |
| tags | 标签集合 |
| source | 来源（hand_written / auto_summary / article / reflection / import） |
| weight | 权重（检索排序用） |
| time_range_start | 时间范围起始 |
| time_range_end | 时间范围结束 |
| review_status | 审阅状态（unreviewed / reviewed / promoted） |
| decay_strategy | 衰减策略标识 |
| is_core | 是否为核心记忆（L0 标记） |
| created_at | 创建时间 |
| last_accessed_at | 最后被检索命中的时间 |

### 7.3 Sensory Snapshot（感知快照）

**定义**

感知快照是系统对自身运行环境或宿主环境在某时刻的采样结果，经规则处理后用于创作上下文与健康分析。

**必备属性**

| 属性 | 说明 |
|------|------|
| id | 唯一标识 |
| source | 来源标记（container / host） |
| sampled_at | 采样时间 |
| cpu_percent | CPU 使用率 |
| memory_percent | 内存使用率 |
| io_read_bytes | IO 读取量 |
| io_write_bytes | IO 写入量 |
| disk_usage_percent | 磁盘使用率 |
| network_rx_bytes | 网络接收量 |
| network_tx_bytes | 网络发送量 |
| load_average | 系统负载 |
| api_latency_ms | API 调用延迟 |
| tags | 标签化结果（如 `high_cpu`、`memory_pressure`、`io_spike`） |
| translated_text | 人格化翻译文本（如"体内灵力激荡不休"） |
| is_in_blind_zone | 是否处于编译盲区（处于盲区的快照不可进入创作上下文） |

**行为约束**

- 感知快照必须与时间点绑定；
- 感知快照必须经过过滤层（标签化 + 盲区检测）；
- 编译盲区内快照不得进入创作上下文；
- 感知仅为辅助，不可凌驾于人格核心锚点之上。

### 7.4 Event（事件）

**定义**

事件是能够刺激数字生命做出响应的触发对象。

**类型**

| 类型 | 说明 |
|------|------|
| `scheduler` | 定时触发 |
| `manual` | 手动触发 |
| `webhook` | 外部 Webhook |
| `folder_monitor` | 目录变化 |
| `system_exception` | 系统异常（极端感知状态） |
| `review` | 回顾触发 |
| `publish` | 发布事件 |

**必备属性**

| 属性 | 说明 |
|------|------|
| id | 唯一标识 |
| event_type | 事件类型 |
| source | 来源描述 |
| raw_payload | 原始载荷 |
| normalized_semantic | 标准化语义描述 |
| auth_status | 鉴权状态（passed / failed / not_required） |
| dedup_key | 去重键 |
| cooldown_status | 冷却状态（ready / cooling） |
| created_at | 创建时间 |
| task_id | 关联的生成任务 ID（可为空） |

**行为约束**

- 所有外部事件必须经过鉴权与标准化；
- 所有事件必须进入统一事件总线抽象；
- 事件本身不直接写文章，而是触发生成任务。

### 7.5 Generation Task（生成任务）

**定义**

生成任务是系统中最重要的运行时状态机实体，代表一次完整的创作尝试。

**必备属性**

| 属性 | 说明 |
|------|------|
| id | 任务 ID |
| trigger_source | 触发源类型 |
| event_id | 触发事件 ID |
| persona_id | 目标人格 ID |
| context_snapshot | 上下文快照（JSON，记录组装时的完整输入） |
| memory_hits | 使用的记忆清单（ID 列表 + 相似度分数） |
| sensory_snapshot_id | 使用的感知快照 ID |
| prompt_summary | Prompt 摘要（结构化记录，非全文） |
| generated_content | 生成结果（原始输出） |
| qa_result | QA 结果（JSON，四项检查详情） |
| retry_count | 重试次数（当前值） |
| max_retries | 最大重试次数（固定为 3） |
| token_input | 输入 Token 数 |
| token_output | 输出 Token 数 |
| cost_estimate | 成本估算 |
| status | 当前状态 |
| started_at | 开始时间 |
| finished_at | 结束时间 |
| post_id | 关联文章 ID（可为空） |

**状态定义与转移规则**

```
                        ┌──────────────────┐
                        │      queued      │
                        └────────┬─────────┘
                                 │
                                 ▼
                        ┌──────────────────┐
                        │preparing_context │
                        └────────┬─────────┘
                                 │
                          ┌──────┴───────┐
                          ▼              ▼
                 ┌──────────────┐  ┌──────────┐
                 │  generating  │  │  failed   │ (上下文组装失败)
                 └──────┬───────┘  └──────────┘
                        │
                  ┌─────┴──────┐
                  ▼            ▼
         ┌──────────────┐  ┌──────────┐
         │ qa_checking  │  │  failed   │ (LLM 调用失败)
         └──────┬───────┘  └──────────┘
                │
          ┌─────┼──────────────────┐
          ▼     ▼                  ▼
  ┌─────────┐ ┌────────────────┐ ┌──────────────────────┐
  │ QA Pass │ │rewrite_pending │ │    circuit_open       │
  │         │ │(retry < max)   │ │(retry >= max, 熔断)    │
  └────┬────┘ └───────┬────────┘ └──────────────────────┘
       │              │                       │
       │              └──▶ generating ──▶ ... │
       │                   (重试循环)          │
       ▼                                      ▼
  ┌─────────────────────────┐         ┌──────────────┐
  │ waiting_human_signoff   │         │  draft_saved  │
  │ (需人工签发时)            │         │  (强制存草稿)  │
  └────────┬────────────────┘         └──────────────┘
           │ 或直接
           ▼
  ┌──────────────────┐
  │ ready_to_publish │
  └────────┬─────────┘
           │
           ▼
  ┌──────────────────┐
  │   publishing     │
  └────────┬─────────┘
           │
     ┌─────┴──────┐
     ▼            ▼
┌───────────┐ ┌──────────────┐
│ published │ │    failed    │ (发布失败)
└───────────┘ └──────────────┘

特殊转移：
  任意非终态 ──▶ aborted （用户手动终止）
```

**行为约束**

- 所有发文行为必须存在对应任务实体；
- 所有重试都属于同一任务上下文内的有限重试（不创建新 Task）；
- 任务状态流转必须可审计（每次转移记录到 AuditLog）；
- 不允许从终态（published / failed / circuit_open / aborted）回退到非终态。

### 7.6 Post / Draft（文章 / 草稿）

**定义**

文章是系统最终产出物，分为草稿与正式发布稿。

**必备属性**

| 属性 | 说明 |
|------|------|
| id | 文章 ID |
| title | 标题 |
| slug | URL 友好标识 |
| front_matter | Hugo front matter（JSON） |
| content_markdown | 正文 Markdown |
| summary | 摘要 |
| status | 发布状态 |
| persona_id | 关联人格 |
| task_id | 来源任务 ID |
| published_at | 发布时间 |
| revision | 修订版本号 |
| publish_target | 发布目标（hugo / wordpress / telegram 等） |
| digital_stamp | 数字印记（ASCII Art / SVG 签名） |
| review_info | 审核信息（审核人、审核时间、审核备注） |
| created_at | 创建时间 |
| updated_at | 最后修改时间 |

**状态定义**

| 状态 | 说明 |
|------|------|
| `draft` | 草稿 |
| `pending_review` | 等待人工审核 |
| `approved` | 审核通过，待发布 |
| `publishing` | 发布中 |
| `published` | 已发布 |
| `publish_failed` | 发布失败 |
| `archived` | 已归档 |
| `aborted` | 已废弃 |

**行为约束**

- 草稿与文章必须共享统一内容模型；
- 已发布文章允许修订，但修订需保留历史（revision 递增）；
- 发布目标抽象于发布器层，不得直接写死业务流程。

### 7.7 Ghost（数字生命迁移包）

**定义**

Ghost 是系统导出的"数字生命整体存档"，用于跨机器迁移与恢复。

**包含内容**

| 项目 | 说明 |
|------|------|
| personas | 人格定义（全量） |
| config | 核心配置（不含敏感密钥，除非用户显式允许） |
| memories_l0 | L0 核心锚点（全量） |
| memories_l1 | L1 阶段摘要（全量） |
| memories_l2 | L2 文章摘要（可配置导出范围） |
| vector_index | 向量索引数据 |
| posts_metadata | 文章历史元数据（不含全文，除非用户选择） |
| health_snapshot | 导出时的健康状态快照 |
| system_version | 系统版本信息 |
| export_checksum | 导出校验信息（SHA-256） |
| manifest | 包清单（内容列表与版本映射） |

**行为约束**

- Ghost 不是整机镜像，不包含容器镜像、Hugo 主题或 Caddy 配置；
- Ghost 必须有版本兼容策略（manifest 中记录版本号，导入时校验）；
- Ghost 默认不导出敏感密钥明文（API Key 等），除非用户在导出时显式允许；
- Ghost 包格式为单个 `.ghost` 文件（实质为带 manifest 的压缩归档）。

### 7.8 AuditLog（审计日志）

**定义**

审计日志记录系统中所有关键操作的不可变痕迹，用于事后追溯与安全审计。

**必备属性**

| 属性 | 说明 |
|------|------|
| id | 唯一标识 |
| timestamp | 操作时间 |
| actor | 操作者（system / user / scheduler / webhook） |
| action | 操作类型（如 `persona.create`、`task.status_change`、`config.update`、`ghost.export`） |
| target_type | 操作对象类型（persona / task / post / memory / config / ghost） |
| target_id | 操作对象 ID |
| detail | 操作详情（JSON，记录变更前后值或关键参数） |
| ip_address | 操作来源 IP（仅用户操作时记录） |
| severity | 严重级别（info / warning / critical） |

**行为约束**

- 审计日志只追加、不修改、不删除（append-only）；
- 以下操作**必须**记录审计日志：
  - 人格创建 / 修改 / 切换 / 删除
  - 任务状态每次转移
  - 文章发布 / 修订 / 归档
  - 配置变更（API Key、域名、预算等）
  - Ghost 导出 / 导入
  - 登录成功 / 失败
  - 熔断触发
  - 预算休眠触发
- 审计日志应有保留策略（如超过 180 天的 `info` 级日志可自动清理，`critical` 级永久保留）。

### 7.9 CostRecord（成本记录）

**定义**

成本记录是对每次模型调用的 Token 消耗与费用估算的结构化跟踪对象。

**必备属性**

| 属性 | 说明 |
|------|------|
| id | 唯一标识 |
| task_id | 关联的生成任务 ID（可为空，如回顾任务） |
| call_type | 调用类型（generation / embedding / reflection / qa_enhanced） |
| model_id | 使用的模型标识 |
| token_input | 输入 Token 数 |
| token_output | 输出 Token 数 |
| token_total | 总 Token 数 |
| cost_estimate | 成本估算（基于用户配置的单价或默认估算） |
| currency | 币种（默认 USD） |
| created_at | 调用时间 |
| response_latency_ms | 响应延迟 |

**行为约束**

- 每次模型调用（无论 LLM 生成还是 Embedding）都必须写入一条 CostRecord；
- 若上游 API 不返回 `usage` 字段，必须基于 Token 估算规则填充（如按字符数估算）；
- 成本记录不可修改，只可追加；
- 预算检查基于 CostRecord 的聚合查询（按日 / 周 / 月汇总）。

## 8. 功能模块与职责边界

以下模块构成系统最终功能全集。每个模块需具备明确职责、实现方式与边界。

> [!NOTE]
> 本章定义的是各模块的**职责范围与约束边界**。第 9 章将对跨模块的关键技术策略进行补充细化，二者是"职责划分"与"策略约束"的互补关系，而非替代关系。

### 模块 1：部署与运行基座

**职责**：提供系统容器化部署、服务编排、数据持久化与统一运行环境。

**功能**：Docker Compose 一键部署；核心服务自动拉起；持久化卷挂载；容器重启恢复；环境变量加载；运行时目录规划。

**实现方式**：
- 采用标准 `docker-compose.yml`
- 所有服务通过容器运行
- 使用 Named Volume 挂载 SQLite、Hugo 内容、站点产物
- 不依赖宿主机 Python / Hugo / Node 安装

**边界**：
- 不引入 systemd 作为核心调度依赖；
- 不要求用户执行复杂宿主机脚本；
- 不使用 MySQL / PostgreSQL 等重量级数据库作为默认存储。

---

### 模块 2：初始化安全锁与后台访问控制

**职责**：在首次启动与日常管理过程中提供最小安全边界。

**功能**：首次启动锁定态；默认凭证登录（`admin / quanzhen`）；强制修改密码；强制完成基础配置（域名 + API 接入）；会话鉴权；操作权限保护；审计日志。

**实现方式**：
- 在 SQLite 中存储初始化状态标志
- 登录后若未完成初始化，强制跳转引导流程
- 使用 JWT（短生命周期 + HttpOnly Cookie）进行会话管理
- 敏感操作要求鉴权上下文验证

**边界**：
- 默认单管理员模型，不做复杂企业级 RBAC；
- 不开放匿名后台写操作；
- 不做 OAuth / SSO 集成（非目标场景）。

---

### 模块 3：模型接入与 AI 配置中心

**职责**：统一管理大模型与 Embedding 能力的接入方式。

**功能**：配置 Base URL / API Key / Model ID；配置 Embedding Endpoint（可复用 LLM 提供商或独立端点）；连通性测试；模型可用性检测；成本参数配置（单价/千 Token）；调用记录。

**实现方式**：
- 使用统一 LLM Adapter（兼容 OpenAI 风格接口）
- 生成能力与 Embedding 能力分别抽象为独立 Adapter
- API Key 加密存储（不以明文写入数据库）
- 配置变更记录到 AuditLog

**边界**：
- 不绑定单一厂商；
- 不默认承担本地大模型推理；
- 不将复杂底层采样参数（Temperature / Top-P / Frequency Penalty）直接暴露给普通用户——通过"大白话"映射层间接控制。

---

### 模块 4：人格系统（Persona Engine）

**职责**：定义和维护数字生命的身份、语言、世界观与感知解释规则。

**功能**：创建人格；编辑人格；启用/停用人格；切换当前默认人格；管理人格 Prompt 片段（身份、世界观、风格、禁忌）；管理感知意象词典；管理人格边界与表达偏好；计算人格稳定度。

**实现方式**：
- 人格结构化存储于 SQLite（非单段文本）
- Prompt 由模板片段拼装（见 §9.2 Prompt 组装策略）
- 感知意象词典作为人格专属资源，随人格切换而变化
- 任务执行时显式绑定人格 ID，记入 Task 和 Trace

**边界**：
- 不以单段超级 Prompt 替代人格结构；
- 不允许多人格混合导致精神分裂式输出；
- 人格修改不应绕过人工确认机制影响核心锚点（L0 记忆）。

---

### 模块 5：记忆系统（Memory Engine）

**职责**：负责长期记忆、短期摘要、向量检索、衰减压缩与记忆回顾。

**功能**：维护 L0/L1/L2/L3 多层记忆；为文本生成 Embedding；进行 Top-K 语义检索；注入关键记忆上下文；记录命中记忆与相似度；执行记忆衰减；执行主动回顾（定时 LLM 驱动）；将摘要提升为锚点（需人工确认）；供 Ghost 导出。

**实现方式**：
- SQLite + sqlite-vec 扩展
- Embedding 由外部兼容 API 提供（用户配置的 Embedding 端点）
- 检索综合考虑：Embedding 相似度 × 时间衰减因子 × 层级权重 × 人格匹配 × 来源可信度
- 定时执行衰减任务（APScheduler）和回顾任务

**边界**：
- 不无上限堆积原始全文向量——L3 定期清理，L2 衰减后压缩；
- 不允许向量检索覆盖或篡改核心锚点（L0 为只读来源）；
- 不把记忆系统做成通用知识库平台。

---

### 模块 6：感知与意象系统（Sensory Engine）

**职责**：从运行环境采集状态，并转化为符合人格设定的情绪与意象输入。

**功能**：采集 CPU / MEM / IO / Disk / Net / Load；支持容器视角（默认，`psutil` 容器内采集）和宿主增强视角（可选，挂载 `/proc`、`/sys`）；过滤内部噪声；定义编译盲区（Hugo 编译前后各 30 分钟）；规则标签化（如 CPU > 80% → `high_cpu`）；人格化翻译（通过人格的 `sensory_lexicon` 词典）；输出感知文本片段；存储感知快照。

**实现方式**：
- 定时采样（APScheduler，如每 5 分钟）
- 先规则层处理（阈值 → 标签），再通过人格词典翻译（标签 → 意象文本）
- 将结果作为上下文片段注入生成 Prompt
- 快照与 Task Trace 建立关联

**边界**：
- 不提供持续监控、实时告警或历史趋势分析等运维级能力；
- 不直接暴露原始数值指标给生成模型——必须经过标签化和翻译；
- 感知不得成为压倒人格核心锚点与记忆的主导输入（Prompt 中占比受限）。

---

### 模块 7：事件驱动内核（Event Engine）

**职责**：统一接收、校验、节流、标准化和分发所有触发事件。

**功能**：定时事件；手动事件；Webhook 事件；文件夹变化事件；系统异常事件；冷却时间锁（Cooldown Period）；去重（Deduplication Key）；幂等控制；事件标准化（统一语义格式）。

**实现方式**：
- 统一事件模型（Event 领域对象）
- Webhook 使用 HMAC 签名或固定 Bearer Token 鉴权
- 文件夹监控基于 `watchdog` / `inotifywait`
- 事件进入统一总线，由调度与编排系统消费
- 冷却时间默认可配（如"同类 Webhook 事件每 30 分钟最多响应 1 次"）

**边界**：
- 不允许外部事件直接越过安全边界触发发布——必须走完整生成链路；
- 不允许未鉴权事件进入主链路（静默丢弃 + 记录审计日志）；
- 不将事件模块扩展为通用工作流平台。

---

### 模块 8：调度与任务编排系统

**职责**：统一管理定时任务、周期采集、回顾任务与发文任务的计划执行。

**功能**：定时发文；定时采样；定时回顾；定时衰减；持久化调度任务；容器重启后恢复；调度冲突处理；任务锁与互斥控制。

**实现方式**：
- 内嵌 APScheduler（`AsyncIOScheduler`）
- JobStore 持久化到 SQLite（`SQLAlchemyJobStore`），容器重启后自动恢复
- 调度只负责触发事件，不替代业务状态机
- 业务结果写入独立任务表（Generation Task）

**边界**：
- 不依赖宿主 systemd / crontab；
- 不允许调度器承担全部业务状态——调度器触发 → 状态机流转 → 业务表记录；
- 不允许多个重叠任务同时争抢发布资源而无锁（见 §9.5 并发控制）。

---

### 模块 9：生成编排内核（Generation Orchestrator）

**职责**：将人格、记忆、感知、事件与模型调用组织成可控的创作链路。

**功能**：选择人格；检索记忆（Top-K）；选择感知快照；解析事件语义；组装上下文；构建 Prompt（结构化拼装）；调用模型生成；记录任务细节；重试与回写。

**实现方式**：
- 明确状态机（Generation Task 状态流转）
- Context Builder 结构化拼接各来源上下文
- Prompt Builder 最终组装完整 Prompt
- 每一步输出中间结果摘要进入 Trace

**边界**：
- 不允许跳过中间记录形成黑盒链路——每步必须有 Trace 输出；
- 不允许隐式散落在多个模块里拼 Prompt——所有 Prompt 构建集中于 Prompt Builder；
- 不允许无限重试（上限 3 次）或越权重写（熔断后不可自动恢复）。

---

### 模块 10：内容质检与熔断系统（QA & Circuit Breaker）

**职责**：在文章输出后执行自动检查、重写控制和熔断保护。

**功能**：长度检查；禁词检查；模板化句式检测；历史重复度检测；风险评级；自动重写；超限熔断；草稿回退；高风险人工签发。

**实现方式**：

| 检查项 | 实现方式 | 是否消耗 LLM Token |
|--------|---------|-------------------|
| 文章长度 | Python `len()` + 字数统计 | ❌ |
| 系统禁词 | 正则匹配预设禁词表 | ❌ |
| 模板化句式（机器味） | 预设"AI 口头禅"正则库（如"首先/其次/总之"高频套用） | ❌（可选 LLM 增强审查） |
| 历史重复度 | 与已有文章的向量相似度对比（复用 sqlite-vec） | ❌（复用已有 Embedding） |

- 默认四重质检**不消耗额外 LLM Token**
- 重写次数严格封顶（`max_retries = 3`）
- 熔断后终止自动链路，强制存入草稿箱并记录原因

**边界**：
- 不以 LLM 审查作为默认必须依赖——LLM 深审为可选增强项；
- 不允许无限自我修复——3 次重试后强制熔断，写入 AuditLog；
- 不允许熔断后仍自动发布——熔断态的 Post 只能进草稿箱。

---

### 模块 11：反完美与残缺感引擎（Anti-Perfection Engine）

**职责**：为数字生命提供受控的波动、不稳定感与偶发的"非完美表达"。

**功能**：根据异常状态触发非常态写作（"走火入魔"模式）；调整句式、节奏、结构、发散度；输出碎片化、意识流或狂气短文；设置冷却与频率上限；自动回落常态。

**实现方式**：
- 将极端感知标签（如 `memory_critical` + `io_spike`）、事件强度、人格参数共同作为触发条件
- 改变输出模板与约束（如强制缩短篇幅、提高 Temperature），而非完全放弃控制
- 仍然走质检和安全链路（禁词检查不豁免）

**边界**：
- 走火入魔模式**不是真失控**——禁词、安全与预算边界仍然有效；
- 连续触发次数封顶（如连续 3 次走火入魔后强制冷却 24 小时），防止长期人格崩坏；
- WebUI 上必须有明确的视觉反馈（如仪表盘高亮/脉冲动画），告知用户"当前处于非常态"；
- 走火入魔的触发与恢复必须记入 AuditLog。

---

### 模块 12：内容管理与人工共生系统

**职责**：提供人工查看、介入、审阅、修订与签发能力。

**功能**：手动发文；草稿箱；历史文章查看与修改；草稿重写；任务日志查看；人工签发；重大设定变更确认延迟；查看生成依据（Trace 详情）。

**实现方式**：
- 后台提供文章视角与任务视角两种浏览模式
- 草稿与任务一一关联，可从任务回溯到草稿，也可从草稿查看任务详情
- 审批通过后触发状态流转控制

**边界**：
- 不做多人实时协同编辑；
- 不做团队内容审批流引擎；
- 人工干预是边界控制，不是替代数字生命主体性。

---

### 模块 13：发布器引擎（Publisher Engine）

**职责**：将文章从系统内部结构转化为外部发布结果。

**功能**：生成 Hugo 所需 Markdown / Front Matter；写入内容目录（`hugo_content` Volume）；触发 Hugo 编译（通知 Sidecar）；管理 slug 与发布时间；预留 WordPress / Telegram / Substack / 小红书适配接口。

**实现方式**：
- 使用 Publisher Adapter 抽象接口
- `HugoPublisher` 为默认主实现——将 Markdown 写入共享 Volume，触发 Sidecar 编译
- 编译任务由 Hugo Builder Sidecar 独立执行
- 发布结果回写文章状态（Post status）
- 新增发布目标只需实现 Adapter 接口，不修改核心编排逻辑

**边界**：
- 不将 Hugo CLI 逻辑散布在业务层——Hugo 相关操作封装在 `HugoPublisher` 内；
- 不在当前系统内承担图床/媒体处理——所有媒体资源只保存外部链接；
- 多平台能力必须经抽象适配，不得污染核心领域模型。

---

### 模块 14：Ghost 迁移系统

**职责**：实现数字生命状态的打包、迁移、恢复与版本兼容。

**功能**：导出 `.ghost`；导入 `.ghost`；校验一致性（checksum）；预览导入内容；解决冲突（如目标系统已有同 ID 人格）；恢复人格、记忆、向量索引与历史元数据。

**实现方式**：
- 定义 Ghost 包格式：`manifest.json` + 分类数据文件 + 向量数据 + checksum
- 提供版本号与 SHA-256 校验值
- 导入前执行兼容性检查与内容预览

**边界**：
- 不等同于整机快照——不含容器镜像、Hugo 主题、Caddy 配置；
- 默认不导出敏感密钥明文（需用户在 WebUI 导出时勾选"包含 API Key"）；
- 不承诺任意跨大版本无损兼容——manifest 中记录版本号，不兼容时给出明确错误信息。

---

### 模块 15：健康度与可观测系统

**职责**：为用户提供数字生命运行状态、人格稳定性、记忆一致性与任务链路可视化。

**功能**：Persona Stability Score；Memory Coherence Score；感知曲线；Token 与成本仪表盘；任务成功率；平均生成耗时；Top-5 记忆命中展示；Trace 查看；结构化日志查看/导出。

**实现方式**：
- 通过规则与统计计算评分（非 LLM 计算，零额外 Token 消耗）
- 关键任务步骤打点（每步写 Trace 记录到 SQLite）
- 结果存入数据库并在 WebUI 可视化
- 可选对接外部日志/Trace 系统（如 LangFuse），但不作为运行必须依赖

**边界**：
- 不使可观测模块重于主业务——评分计算必须轻量；
- 不要求依赖外部 observability 平台才能正常运行；
- 分数是辅助解释工具，不应被视为绝对真理——用于"帮助用户理解数字生命状态"。

---

### 模块 16：成本控制与休眠系统

**职责**：控制模型调用成本，防止无限消耗。

**功能**：记录 Token 消耗（CostRecord）；估算调用成本；按日/周/月聚合；预算阈值告警；超限自动休眠；恢复与解锁；降级模式。

**实现方式**：
- 所有模型调用写 CostRecord（见 §7.9）
- 根据用户配置的单价规则估算成本；若用户未配置单价，则使用默认估算公式
- 达到日预算阈值后自动阻断自动链路（定时发文暂停），WebUI 显示休眠状态
- 管理员可手动恢复（解除休眠）
- 降级模式：预算接近阈值时，自动减少发文频率或缩短文章长度

**边界**：
- 不允许自动生成链路绕过预算阈值检查——Budget Check 在链路最前端；
- 不因休眠损坏已有任务数据——休眠只阻断新任务，不中断已在执行的任务；
- 若上游 API 不返回 usage 字段，必须支持基于字符数的估算模式。

## 9. 关键实现策略

> [!NOTE]
> 本章是对第 8 章模块边界的**跨模块策略补充**，定义若干必须在全局层面统一的技术约束。

### 9.1 存储策略

- 默认数据库：SQLite（单文件 `quanzhen.db`）
- 启用 WAL 模式以改善并发读写性能
- 调度任务状态、业务任务状态、文章、记忆、日志、配置、成本记录**分表存储**
- 向量能力使用 sqlite-vec 扩展
- 所有关键业务状态均需可迁移、可备份、可导出
- 数据库文件存储在 `app_data` Volume 中

**边界**：
- 不默认引入 MySQL / PostgreSQL；
- 不允许将核心状态散落在不可导出的缓存中（如仅存于内存的 dict）；
- 不允许过度依赖内存态作为唯一真实状态源。

### 9.2 Prompt 组装策略

Prompt 由以下结构化片段拼装，顺序固定：

| 顺序 | 片段 | 来源 |
|------|------|------|
| 1 | 系统行为约束 | 固定模板 |
| 2 | 人格设定（身份 + 世界观 + 风格 + 禁忌） | Persona 实体 |
| 3 | 当前触发事件语义 | Event 标准化结果 |
| 4 | 感知意象文本 | Sensory Snapshot 翻译结果 |
| 5 | 命中记忆上下文（Top-K） | Memory 检索结果 |
| 6 | 输出格式要求（标题、长度、结构） | Persona 偏好 + 全局配置 |

- 每个片段来源必须可追踪（记入 Trace）
- 需要保存最终 Prompt 的**结构化快照**（记录各片段来源 ID 和实际注入的文本摘要），而非全文存储（节省空间）

**边界**：
- 不允许所有逻辑堆入单段长 Prompt——必须分段拼装；
- 不允许不同模块偷偷拼接 Prompt 导致不可解释——所有拼接逻辑集中于 Prompt Builder。

### 9.3 记忆检索策略

语义检索必须综合以下维度：

| 维度 | 权重来源 | 说明 |
|------|---------|------|
| Embedding 相似度 | sqlite-vec 计算 | 语义匹配基础分 |
| 时间衰减 | 基于 `created_at` 和层级衰减参数 | 越新的记忆分数越高 |
| 层级权重 | L0 > L1 > L2 > L3 | 核心锚点优先 |
| 人格匹配 | `persona_id` 过滤 | 只检索当前人格的记忆 |
| 来源可信度 | `source` 字段 | 手写 > 回顾提升 > 自动摘要 |

- 检索结果只选 Top-K 注入（K 默认为 5，可在 WebUI 高级模式调整）
- 每次生成后记录命中记忆 ID 与相似度分数
- 长期运行必须有衰减（定时任务降低 weight）与主动回顾（LLM 生成摘要）

**Embedding 降级策略**：

| 优先级 | 方案 | 触发条件 |
|--------|------|---------|
| 1 | 用户配置的 Embedding 端点 | 正常模式 |
| 2 | 复用 LLM API 的 Embedding 能力 | 用户未单独配置 Embedding 时尝试 |
| 3 | 关键词匹配降级 | Embedding API 不可用时的应急方案 |

- 降级发生时记录 AuditLog（severity: warning）
- 降级模式下关闭向量相似度排序，改为基于标签 + 时间的简单匹配

**边界**：
- 不允许把所有记忆原样塞回上下文——严格 Top-K 截断；
- 不允许只按向量相似度单维排序——必须综合多维加权。

### 9.4 感知翻译策略

翻译流程：

```
原始指标 (float)
  → 规则层：阈值判定 → 结构化标签 (string[])
    → 翻译层：人格意象词典查找 → 人格化文本片段 (string)
```

- 规则层由 Python 代码硬编码（如 `cpu > 80 → "high_cpu"`）
- 翻译层由 Persona 的 `sensory_lexicon` 驱动（同一 `high_cpu` 标签在不同人格下产生不同文本）
- 对 Hugo 编译等内部构建行为设盲区（前后各 30 分钟），盲区内快照标记 `is_in_blind_zone = true`
- 感知文本只作为 Prompt **上下文片段**注入，不作为系统指令

**边界**：
- 不允许直接将 raw metrics 大段注入模型——必须经过标签化和翻译；
- 不允许感知翻译完全交给 LLM 现编——必须先有规则基础，LLM 只做可选润色。

### 9.5 并发控制与任务锁策略

**全局并发模型**：

在 1C1G / 2C2G 的目标硬件上，系统采用**单活任务**模型：

| 约束 | 说明 |
|------|------|
| 生成任务至多 1 个活跃 | 同一时刻只有一个 Generation Task 处于非终态 |
| Hugo 编译至多 1 个并行 | 编译任务串行执行，排队等待 |
| 发布锁 | 发布过程（写文件 + 触发编译）持有独占锁 |
| 感知采样与生成互不阻塞 | 采样为独立轻量 goroutine，不持锁 |

**锁实现**：
- 使用 SQLite 行级锁（或应用层 `asyncio.Lock`）实现任务互斥
- 所有生成任务必须存在幂等键（trigger_source + event_id + timestamp 复合键）
- 同类事件需具备去重策略（Dedup Key + 冷却间隔）
- Webhook 需防重放（时间窗 + nonce）
- 重试必须挂载在同一任务上下文下（不创建新 Task）

**边界**：
- 不允许定时、手动、Webhook 之间互相踩踏生成同一篇内容；
- 不允许并发发布破坏 Hugo 内容目录一致性。

### 9.6 质检与重写策略

- 默认先规则审查（四重质检），再决定是否重写
- 重写次数严格封顶（`max_retries = 3`）
- 每次重写都需记录原因（写入 Task 的 `qa_result` JSON）
- 若持续失败则熔断（Task 状态 → `circuit_open`），强制存入草稿箱
- 高风险内容（质检风险评级为 `high`）转入人工处理（Task 状态 → `waiting_human_signoff`）

**边界**：
- 不允许自动链路无限循环——3 次即硬性上限；
- 不允许重写丢失任务上下文——重写复用同一 Task，`retry_count` 递增。

### 9.7 域名与 HTTPS 策略

- 后台可配置域名（WebUI "站点设置" 页面）
- 只有在域名 DNS 解析正确指向当前服务器 IP 时启用 HTTPS
- 域名配置变更后，系统更新 Caddy 配置并重载
- 必须提供域名启用状态与失败原因反馈（WebUI 显示 + AuditLog 记录）
- Caddy 接管证书申请与续期（ACME），无需用户手动操作

**边界**：
- 不做完全黑盒、不可排障的魔法切换——域名启用失败需给出明确原因（如"DNS 未解析"、"端口未开放"）；
- 域名启用失败不得破坏系统其他能力——系统可以在无域名（纯 IP）模式下完整运行。

### 9.8 冷启动策略

全新部署时，记忆库为空、感知无历史基线。系统须定义明确的冷启动行为：

| 条件 | 行为 |
|------|------|
| 无 Persona 存在 | 阻止生成任务，提示用户创建至少一个人格 |
| Persona 存在但无 L0 记忆 | 仅使用人格自身的 `identity_setting` 和 `worldview_setting` 生成——效果等同于 L0 锚点 |
| 无 L1/L2/L3 记忆 | 跳过记忆检索步骤，Prompt 中记忆片段为空 |
| 无感知历史 | 使用当前感知快照（如果有）或跳过感知片段 |
| 无 Embedding API 配置 | 以关键词匹配降级模式运行，记忆检索精度降低但不阻断运行 |

**冷启动引导**：
- 首篇文章生成后，自动创建 L2 摘要记忆
- 首次回顾任务执行后，自动创建 L1 摘要
- 系统在冷启动阶段的生成任务 Trace 中标记 `cold_start = true`

### 9.9 错误恢复策略

| 故障场景 | 恢复策略 |
|---------|---------|
| 生成过程中容器崩溃重启 | APScheduler JobStore 从 SQLite 恢复；处于非终态的 Task 标记为 `failed`（带原因 `container_restart`）；不自动重试已中断的 Task |
| Hugo 编译中容器重启 | Hugo Builder 重启后重新编译 `hugo_content` 全量内容；编译中间产物不保证原子性——因此 `hugo_public` Volume 在编译完成前不切换 |
| LLM API 调用超时 | 单次调用超时视为失败，触发重试（在 `max_retries` 内）；连续超时 N 次后记录 AuditLog 并通知用户（WebUI 告警） |
| SQLite 写入失败 | 启用 WAL 模式降低写冲突概率；写入失败的操作回滚并重试一次；若仍失败则 Task 标记为 `failed` |
| Embedding API 不可用 | 降级到关键词匹配模式（见 §9.3）；记录 AuditLog（warning）；不阻断发文链路 |

**通用原则**：
- 宁可丢失一次发文机会，也不可产生状态不一致；
- 所有异常恢复都必须有 AuditLog 记录；
- 容器重启后，系统自动执行一次"状态自检"：扫描所有非终态 Task，将悬挂任务标记为 `failed`。

## 附录 A：术语表

| 术语 | 定义 |
|------|------|
| **核心锚点（Core Anchor）** | L0 层记忆，包含人格的根基性设定（身份、世界观、长期关系），永不自动衰减 |
| **感知快照（Sensory Snapshot）** | 系统对运行环境在某时刻的结构化采样结果，包含原始指标、标签和人格化翻译文本 |
| **编译盲区（Build Blind Zone）** | Hugo 编译前后各 30 分钟的时间窗口，期间的感知快照标记为不可信，不注入创作上下文 |
| **Ghost** | 数字生命的整体导出存档包（`.ghost` 文件），用于跨 VPS 迁移与恢复 |
| **走火入魔（Anti-Perfection Mode）** | 当系统检测到极端环境状态时触发的非常态写作模式，提高发散度，输出碎片化内容 |
| **人格意象词典（Sensory Lexicon）** | 绑定在人格上的映射表，将感知标签（如 `high_cpu`）翻译为人格化文本（如"真气激荡"） |
| **生成任务（Generation Task）** | 一次完整的创作尝试，从触发到发布/失败的全链路状态机实体 |
| **Trace** | 生成任务中每一步骤的结构化中间记录，用于事后追溯 |
| **熔断（Circuit Break）** | 当重试次数达到上限时强制终止自动链路，将产物存入草稿箱 |
| **冷却时间（Cooldown Period）** | 同类事件之间的最小间隔时间，防止高频触发 |
| **大白话交互** | 将内部 AI 参数（如 Temperature）转化为用户可理解的日常用语（如"情绪波动程度"） |
| **数字印记（Digital Stamp）** | 每篇文章生成时自动创建的唯一 ASCII Art 或 SVG 签名，作为内容身份标识 |
| **Publisher Adapter** | 发布器的抽象接口，允许不同发布目标（Hugo / WordPress / Telegram）实现统一调用契约 |
| **LLM Adapter** | 大模型调用的抽象接口，兼容 OpenAI 风格 API，屏蔽不同厂商差异 |
| **记忆衰减（Memory Decay）** | 随时间推移降低记忆权重的机制，L2（90天）、L3（30天）自动降权，L0 永不衰减 |
| **主动回顾（Active Reflection）** | 定时由 LLM 回顾近期记忆并生成阶段摘要（L1），可被人工审核后提升为核心锚点（L0） |
| **CostRecord** | 每次模型调用的 Token 消耗与费用估算的结构化记录 |
| **AuditLog** | 系统关键操作的不可变追踪日志，只追加不修改 |
